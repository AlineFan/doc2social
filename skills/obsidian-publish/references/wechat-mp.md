# 微信公众号长文格式规则

> 生成公众号正文时读这个文件。
> **2026-06-18 调整**：本 skill 输出 **markdown**，不输出 HTML。HTML 转换交给下游工具（mdnice / md2weixin-core / 公众号编辑器辅助插件）。

## 0、为什么不直接输出 HTML

| 维度 | markdown | HTML |
|------|----------|------|
| 用户可读性 | ✅ 直接阅读 | ❌ 标签噪音 |
| 用户改字 | ✅ 改 markdown 就行 | ❌ 容易破坏样式 |
| 样式灵活度 | ✅ 套不同主题不同效果 | ❌ 样式锁死 |
| 下游工具链 | ✅ mdnice / md2weixin-core 都吃 | ❌ 转换好后是终态 |

**结论**：本 skill 永远输出 markdown。HTML 转换是下游的事。

## 一、公众号编辑器的最终接受格式（背景知识）

公众号编辑器最终吃的是 **inline-style HTML**（不吃 markdown、不吃外部 CSS、不吃 class）。但这个转换由下游工具做：

- **手动路径**：复制 markdown → mdnice.com → 选主题 → 复制富文本 → 公众号编辑器粘贴
- **自动路径**（Phase 3）：用 `md2weixin-core` 的 `getHtml({markdown, theme})` 函数

我们生成的 markdown 必须保证经过这两个路径转换后效果好。

## 一·5、公众号长文整体结构模板

> 借鉴 khazix-writer 第十一层 + 去掉固定尾部 / dbs 风格调校。
> **不是死模板**——是骨架，按选定的文章原型（见 `references/article-archetypes.md`）填充。

```
【开头】感性切入，从具体事件 / 场景开始，迅速建立情绪
  - 不要「在当今...的时代」式宏大开头
  - 用「叙事启动 / 荒诞事实 / 热点破题 / 好奇心驱动」四种之一
  ↓
【背景铺垫】简要科普，聊天式而非教科书式
  - 不要「下面我来介绍」「首先需要了解」
  - 知识是「聊着聊着顺手掏出来」的
  ↓
【核心内容】按原型展开 3-5 个板块，每块有：
  - 1 个明确观点
  - 至少 1 个具体场景 / 人物 / 对话 / 数据支撑
  - 私人视角连接（「我自己也是这样」）
  - 1 句扣主线的话，把偏出去的拉回来
  - 板块间用口语化转场（「说到这个」「回到 xxx 这块」），**不用 markdown 小标题**
  ↓
【升华】（仅"现象解读型"原型必须有）
  从具体事件拉到更大的文化 / 哲学 / 历史参照
  不是论文式总结，更像「聊着聊着自然想到了」
  ↓
【收尾】挑一种自然收法（不要套话）：
  - 引用收尾：用别人的一句话作结
  - 哲思余韵：一个短句留白
  - 行动呼吁：鼓励读者去做一件事（**dbs 风格首选**）
  - 信念宣言：表达对未来的坚信
  - 回环呼应：回到开头意象但视角已变

【不要的】
  - ❌ 「既然看到这里了，点赞在看转发三连」（khazix 风格的固定尾部，dbs 反对）
  - ❌ 「希望对你有帮助」
  - ❌ 「我们下次再见」
```

### 字数 vs khazix

khazix 写卡兹克风格用 4000-8000 字。**dbs 风格更克制：1500-3000 字最佳**。
- 低于 800 字 → 显"水"
- 高于 4000 字 → dbs 风格的张力会被冲淡

按 dbs 哲学：「投入精力 × 对内容有正确理解 = 内容好」。**少而精胜于多而水**。

## 二、markdown 输出骨架

```markdown
{首段：钩子段，1-2 句话引出核心冲突。**不要**用"亲爱的家人们"开场}

{第二段：交代背景或一句金句引用}

> {引用块：原作者金句或核心观点}

{第三段：自然过渡到第一个小标题}

## 一、{小标题：观点 / 反差}

{段落 1：核心论点}

{段落 2：展开 / 例子}

## 二、{小标题}

{...}

## 写在最后

{收束 + 一个具体行动}
```

**关键**：用 markdown 的 `##` `###` 当小标题、`>` 当引用、`**bold**` 加粗。下游工具会按主题渲染样式。

## 三、markdown 写作规范

- **段落短**：每段 12-30 字最佳，移动端读得快
- **每 2-3 段插一个引用块或小标题**，控制节奏
- **强调用 `**bold**`**，不要用颜色 / emoji 来"强调"
- **小标题层级**：`##` 用于章节、`###` 用于子项。不要超过 3 层嵌套
- **列表**：有序列表（`1. 2. 3.`）用于有顺序的事物；无序（`-`）用于平行罗列
- **代码块**：用三个反引号包围，标注语言（` ```js ` / ` ```python `）
- **引用块**：原作者金句、关键观点。一篇文章 2-4 个引用块最佳
- **不要居中**：公众号读者习惯左对齐，markdown 本身也没有居中
- **不要加大段彩色文字**：markdown 不直接支持，且效果 low

## 四、Gotchas 清单

- **图片占位**：原文有图的地方写 `![图：图片描述](placeholder)` 或 `<!-- 图：图片描述 -->`。下游粘贴时图片会丢，必须手动上传到公众号素材库后插入
- **代码块**：用三个反引号 + 语言标识，下游工具会给代码高亮主题。**不要**自己写 inline HTML 模拟代码块
- **强调用粗体不用颜色**：`**重点**` 是 markdown 标准。不要用 inline HTML 的 `<span style="color:red;">`
- **不要 `<br>` 控制间距**：用 markdown 空行
- **不要"亲爱的家人们"开场**：dontbesilent 文字洁癖原则
- **markdown 中**：避免脚注、定义列表等 mdnice / md2weixin-core 不支持的扩展语法
- **链接**：用 `[文字](URL)`，下游会自动处理。不需要预先缩短

## 五、标题生成（公众号专属）

公众号标题逻辑跟小红书不一样：
- **不要用"3 个秘诀"这种**，公众号读者反感感
- **优先用悬念 + 反差**：「我做了一年内容，发现一件反直觉的事」
- **不要超过 64 个字符**（公众号限制）
- **3 个标题**给用户选，标注每个的"点击劫持"程度（悬念 / 反差 / 数字 / 名人）

## 六、字数建议

- **公众号长文**：1500-3000 字最佳。低于 800 字读者觉得"水"，高于 5000 字读者跳出
- **不够字数硬凑**比"短而精"更差，宁可少写

## 七、最后输出前自检

- [ ] 输出是 markdown（不是 HTML）
- [ ] 没有 inline HTML 标签（`<span>` `<div>` 等）
- [ ] 图片都用 markdown 占位（`![图：xxx](placeholder)` 或注释）
- [ ] 没有外链广告 / 无效跳转
- [ ] 段落长度合理（12-30 字/段）
- [ ] 引用块 2-4 个
- [ ] 小标题层级不超过 3 层
- [ ] 没有"亲爱的家人们"开场
- [ ] 标题有钩子（不剧透 + 有反差）
