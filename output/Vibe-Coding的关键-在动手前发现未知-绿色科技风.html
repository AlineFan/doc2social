核心概念：地图不是领土

THE MAP IS NOT THE TERRITORY

文章来自于 Claude Code 团队的工作人员。虽然写的是和 Claude Fable 协作的过程，但"让 AI 寻找业务盲点、头脑风暴、采访、参考代码"这些思路是通用的。

瓶颈不是模型能力，而是你能不能把自己想要什么讲清楚。文章给出了一套完整工作流：动手前扫盲区 → 动手中记笔记 → 动手后测试。

用 Claude 写代码，你给它的是"地图"：你的 prompt、技能、上下文。真正要干活的地方是"领土"：代码库、真实世界的约束。地图和领土之间的差距，就是未知。

Claude 碰到未知的时候，只能猜。未知越多，猜错越多。Claude Fable 是第一个让作者感受到"模型够强了，但活干得好不好取决于你能不能把未知找出来"的模型。

光提前规划还不够。有些未知要到写代码的时候才冒出来，有些未知甚至会告诉你：你一开始要解决的问题就不对。

四种未知

FOUR TYPES OF UNKNOWNS

作者把"未知"拆成四类：

我知道
我不知道
我知道的东西
Known Knowns
比如我写进 prompt 里的需求
Known Unknowns
我还没想清楚，但知道自己没想清楚
我不知道的东西
Unknown Knowns
太显然了我不会写下来，但看到了就知道
Unknown Unknowns
我完全没考虑过的事，甚至不知道"好"能有多好

厉害的 agentic coder 未知少。他们清楚自己要什么，同时熟悉代码库和模型行为。但他们也会主动假设未知存在。减少和规划未知，就是 agentic coding 的核心技能。

03

PART

给 Claude 指令的平衡

BALANCING INSTRUCTIONS

太具体 → Claude 会死板执行，即使该转弯也不转
太模糊 → Claude 会按行业"最佳实践"猜，但可能不适合你的场景

你没处理未知的时候，两头都翻车：不知道路上哪里有坑，也不知道哪里该让 Claude 自己绕路。

所以关键不是写更好的 prompt，而是让 Claude 帮你发现未知。它能快速搜索代码库和互联网，在很多话题上比你懂得多，也能比你更快从失败中迭代。

最重要的是：告诉 Claude 你的起点在哪。你现在想到哪了、你对这个问题和代码库有多少经验、让它像搭档一样跟你一起想。

动手前：五种方法找未知

FIVE METHODS TO FIND UNKNOWNS

01

盲区扫描（Blind Spot Pass）

你在写一个新功能，或者做一个不熟悉的领域，大概率有大量 Unknown Unknowns。你不知道该问什么问题、什么是"好"、以前做过什么、哪里有坑。

直接让 Claude 帮你找盲区。作者喜欢用"blind spot pass"和"unknown unknowns"这两个词。

示例 prompt：

"我要加一个新的 auth provider，但我不了解这个代码库的 auth 模块。帮我做一次盲区扫描，找出我的 unknown unknowns，让我能写出更好的 prompt。"

"我不懂调色，但我需要给这个视频调色。教我发现自己关于调色的 unknown unknowns。"

02

头脑风暴和原型（Brainstorms & Prototypes）

当你有很多 Unknown Knowns，那种"我看到了才知道我要什么"的东西，让 Claude 先做原型。在原型阶段把 Unknown Knowns 找出来，比在写代码的时候才发现便宜得多。小的需求变动可能导致完全不同的实现，而且让 AI 回退之前的改动更难。

示例 prompt：

"我要一个数据 dashboard，但我没有审美也不知道能做到什么。做一个 HTML 页面，给我 4 种完全不同的设计方向，我来挑。"

"先别动真代码，用一个 HTML 文件模拟新的编辑器工具栏，用假数据。我想先看布局再动真 app。"

03

采访（Interviews）

头脑风暴完了还有未知？让 Claude 采访你。给它你的问题背景，让它一个问题一个问题问你。

"一个一个问我所有模糊的地方，优先问那些'我的回答会改变架构'的问题。"

04

参考代码（References）

有时候你描述不出你想要的东西：可能你没那个词汇，可能太复杂说不完。最好的参考是源码。如果你有一个库实现了你想要的行为，或者一个你喜欢的设计组件，直接让 Claude 去读那个文件夹，哪怕语言不同。这比截图提供的信息丰富得多。

"vendor/rate-limiter 里这个 Rust crate 实现了我想要的退避行为。读它，然后在我们的 TypeScript API client 里重新实现同样的语义。"

05

实施计划（Implementation Plans）

觉得准备好了，让 Claude 写一份实施计划给你审。重点放在最可能变的部分：数据模型、类型接口、用户体验流程。机械重构扔到最后。

"写一份实施计划，先列我最可能改的决定：数据模型变更、新的类型接口、所有面向用户的部分。机械重构放最后，那部分我信你。"

05

PART

动手中：实施笔记

IMPLEMENTATION NOTES

不管规划多充分，Unknown Unknowns 总会出现。agent 可能在工作中发现需要换条路。让 Claude 维护一个临时的 implementation-notes.md，记录它做的每一个偏离计划的决定。

开一个新 session，把之前的产物（spec 文件、原型）传进 prompt。干净的上下文窗口 + 之前的所有信息。

"维护一个 implementation-notes.md。如果你碰到边界情况需要偏离计划，选保守的方案，记在 'Deviations' 下面，然后继续。"

PART

动手后：两种方法验证

TWO METHODS TO VALIDATE

01

Pitch 和说明文档

做完一个东西，最难的部分往往是让别人理解并批准。把原型、spec、实施笔记打包成一个文档，方便评审的人快速理解。

"把原型、spec 和实施笔记打包成一个文档，我直接丢 Slack 拉对齐。开头放 demo GIF。"

02

测验（Quizzes）

长 session 结束，Claude 可能做了很多你没意识到的事。光看 diff 不够。让 Claude 给你出一份关于这次改动的测验。测验全对了才 merge。

"我想确保自己理解这次改动的所有内容。给我出一份 HTML 报告，包含上下文、直觉、做了什么，底部加一个我必须通过的测验。"

实战案例：Fable 发布视频

CASE STUDY: FABLE LAUNCH VIDEO

作者用这套方法，纯靠 Claude Code 剪完了 Fable 的发布视频。他和 Claude 的协作过程：

1. 先搞清楚能做什么

他知道 Claude 能用代码编辑视频和做转录，但不确定精度够不够。让 Claude 解释 Whisper 转录的原理，确认能不能用 ffmpeg 精准切掉"嗯""啊"和长停顿。

2. 做原型验证

他想让 UI 跟说话节奏同步，不确定是否可行。让 Claude 用 Remotion + 转录数据做一个原型视频看效果。

3. 发现盲区

视频画面看起来偏灰。他知道这是调色问题，但不懂调色。第一版让 Claude 出了几个变体来挑，结果发现自己不知道什么叫"好"的调色。于是反过来让 Claude 教他调色，先找盲区再动手。

核心结论

KEY TAKEAWAY

模型越强，你用对方法能做到的越多。

一个长任务回来结果不对，大概率不是模型的问题——是你需要花更多时间定义未知，或者写一份能让你和 Claude 一起适应未知的实施计划。

每一个头脑风暴、原型、采访、参考代码，都是在变贵之前找到你"不知道的事"的便宜方式。

所以下次开工前，先让 Claude 帮你找未知。
