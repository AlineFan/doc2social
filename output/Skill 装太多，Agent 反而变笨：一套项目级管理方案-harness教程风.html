skill 太多会带来 噪声、误触发、上下文占用

解决方法：skill 统一管理，项目级按需软链 skill

为什么 skill 还没调用，就已经开始占上下文了

Agent 在工作时有一个上下文窗口，你可以把它想象成 claudecode /codex 的办公桌，办公桌的桌面大小 （上下文窗口）是有限的。

每次开启一个会话，用户发出第一条消息，harness 就会给模型塞一堆信息，其中就包括 skiLL 的列表和摘要（skill name +descriptions）。

以 claude 为例，启动+ 用户发出第一条消息时，harness 和 claude 模型之间的数据交互


用户
  ↓
Claude Code harness
  ├─ 【1】读取 CLAUDE.md
  ├─ 【2】读取 auto memory
  ├─ 【3】发现的 skill列表（按需加载，本次发出的是name+descriptions） / commands / hooks / MCP
  ├─ 【4】处理权限与工具调用
  └─ 把用户消息和以上模块一起，组装上下文发给模型
  ↓
Claude 模型
  ├─ 判断是否需要某个 skill
  ├─ 决定是否调用工具
  └─ 生成回复 / 工具调用
  ↓
Claude Code harness 执行并回传

skill 列表 中的 name + description


- adapt: Adapt designs to work across different screen sizes...
-  animate: Review a feature and enhance it with purposeful animations...
-  animejs: Anime.js adapter patterns for HyperFrames...
- audit: Perform comprehensive audit of interface quality......
- 【100个 skill 的名称+描述，在用户发出第一条消息时，就和用户消息组装一起，发给了模型】

如果我们安装了 100 个 skill，放在全局。

即使这一次会话中只用到了 1 个 skill，这 100 个 skill 的列表和摘要 ，也会在会话之初，就由 harness 摆在了模型的工作台上。

虽然单个 skill 的内容不多，但架不住量大。

用不到的 skill 会默默占据上下文窗口，消耗 token 。

随着会话变长，agent 会更容易"忘"你前面说过的话。

  • 旧信息离当前轮次越来越远
  • 新信息不断堆上来
  • 上下文窗口有限
  • 所以前面的关键信息更容易被噪声淹没

为了方便理解，我们可以用数字做示意：


第 1 轮
- harness 注入：1111
- 用户消息：2222
- 模型本轮看到：1111 + 2222

第 2 轮
- 模型上一轮回复：3333
- 模型本轮可追溯的上下文近似变成：1111 + 2222 + 3333

第 3 轮
- 用户新消息：4444
- 模型本轮看到：1111 + 2222 + 3333 + 4444

第 4 轮
- 模型新回复：5555
- 模型本轮可追溯的上下文近似变成：1111 + 2222 + 3333 + 4444 + 5555

如果会话继续增长，前面的 1111、2222 会越来越"远"。
如果 1111 本身就很臃肿，比如里面混了大量的 skill 描述、工具菜单、无关规则，真正重要的部分就更容易被冲淡。

当然，实际上下文并不一定是"原封不动无限累加"；不同 agent 可能会做裁剪、压缩、摘要或分层加载。 但从最终结果是一样的：会话越长，早期信息越容易失真、弱化或被挤出窗口

此外，skill 太多，还会让 agent 在该调用 A 的时候用了 B，在该调用的时候不调用，或者在不该调用的时候误触发。这个也很好理解，桌面越乱，找东西就越难。

怎么解决

有些 skill 使用频次不高，删除肯定不合适，但不删每次加载又都很浪费。

baoyu 大佬提供了一个思路：skil 放到一个仓库统一管理端，claude / codex 新建项目时，按需从全局 skill 仓库，为单个 skill 建一个软链接。

可以把软链接理解成"快捷方式"

项目里看起来像有这个 skill，实际上文件本体仍然只保存在统一仓库里。

这样做的好处是：

  1. 原件更新一次，所有项目同步生效
  2. 项目里改 skill，本质上改的是原件
  3. 不会在不同项目里出现多份 skill 副本，避免版本漂移

1 skill的位置

要理解这个，首先以 claude 、codex 为例，skill 有 2 个位置，

一个是用户级，所有项目都能看到
【claude code】~/.claude/skills/
【codex】~/.codex/skills/

一个是项目级，只有这个项目才能看到
~/project01/.agents/skills

2 具体操作

【以下软链操作，我们可以通过自然语言，让 AI 去做】

建一个文件夹 A，把所有的 skill 保存在文件夹中。这些 agent 的存储路径就是 ：
~/A/skills/skill01
~/A/skills/skill02
~/A/skills/skill03
~/A/skills/skill04

当我们要新建项目 project02，需要用到 skill02 和 skill04，我们就可以：

  1. 创建 project02，项目地址是：~/project02
  2. 在 ~/project02 里建 skill ：~/project02/.agents/skills/
  3. 然后把 skill02 和 skill04 链进去：
    1. ln -s ~/A/skills/skill02 ~/project02/.agents/skills/skill02
    2. ln -s ~/A/skills/skill04 ~/project02/.agents/skills/skill04
  4. 如果使用的是 claude code，还需要给 claude 补一层入口 ：
    1. 新建 ~/project02/.claude/skills/
    2. ~/project02/.claude/skills/ 指向 ~/project02/.agents/skills

这样处理后：

项目内的路径 实际指向的位置（原件）
~/project02/.agents/skills/skill02
~/project02/.claude/skills/skill02		 ~/A/skills/skill02
~/project02/.agents/skills/skill04
~/project02//.claude/skills/skill04		 ~/A/skills/skill04

这样，project02 里的 skill02、skill04 指向 A 文件里 skill02、skill04
在 project02 中改 skill02，改动的源件，电脑中 skill 版本永远保持统一

到这里，这套 skill 管理方式的核心就很清楚了：

  • skill 原件只保存一份
  • 每个项目只按需挂自己需要的 skill
  • 不用复制，不怕版本漂移
  • 哪个项目缺什么，就给哪个项目补什么
  • 原件修一次，所有项目一起受益

如此一来，agent 每次开工时，我们就能给它提供工具齐全，没有杂物的桌面。

桌面越干净，模型越容易把注意力放在当前任务上。任务自然也就完成的越好。
